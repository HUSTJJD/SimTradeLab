### 1.2 质量过滤（每 20 个交易日刷一次）

| 指标 | 阈值 | 说明 |
|---|---|---|
| 上市时长 | ≥ 365 天 | 排除新股，让 250 日 RSI/750 日回撤窗口可用 |
| 资产负债率 | ≤ 60% | 排除高杠杆（周期股最致命的就是负债端） |
| 商誉占净资产 | ≤ 30% | 排除并购雷 |
| 20 日均成交额 | ≥ 5000 万 | 排除流动性差的小票 |
| **硬止损黑名单**（v1.5 新增）| 1 年内被硬止损 ≥ 2 次 → 隔离 1 年 | 防止同一只股反复抄到刀口 |

> 用 `context.previous_date` 查询，规避聚宽 `avoid_future_data=True` 的"15:00 后才允许查当日市值"限制。

### 1.3 底部多因子打分（4 因子加权 → 0-100 分）

| 因子 | 权重 | 计算 | 满分含义 |
|---|---|---|---|
| **PB 5 年逆向分位** | 0.35 | `(1 - 当前PB在5年中的分位) × 100` | PB 处于 5 年最低 |
| **距前高回撤** | 0.25 | `min(回撤 / 50%, 1) × 100` | 从 750 日高点回撤 ≥50% |
| **OCF / 市值** | 0.20 | `min(ratio / 10%, 1) × 100` | 经营现金流 / 市值 ≥10% |
| **14 日 RSI 逆向** | 0.20 | `100 - RSI(14)` | RSI ≤ 0（极度超卖） |

**触发规则**：综合分 ≥ **70** 持续 **5 个交易日** → 启动 T0；分数低于 65 时退出"持有评估"。

> v1.0 用 250 日 RSI，因子失效；v1.2 改为经典 14 日 RSI 后早年信号显著增多。

### 1.4 分批建仓（30% / 30% / 40%）

```
T0  分数达标 5 天 → 买入底仓 30%        (phase: IDLE → BUILDING)
T1  T0 后跌 ≥8% → 买入底仓 30%
T2  T0 后跌 ≥16%（左侧）or 30 天后涨幅 ≥5%（右侧） → 买入底仓 40%
                                        (phase: BUILDING → GRID_RUNNING)
```

**T0 防幽灵机制（v1.2 引入）**：
- 单窗口最多 1 次下单尝试
- 资金不足 / 涨停一字 / 科创板拒单 / 委托失败 → **立即禁用**，等分数自然回落到 65 以下才能重新累计 5 日信号
- 进入 BUILDING 后清零 `consecutive_signal_days`

### 1.5 动态网格

```
步长 = clamp(1.5 × ATR(20) / last_price, [2.5%, 5%])
买阈值 = 步长 × 0.83        ← 更早买
卖阈值 = 步长 × 1.17        ← 更晚卖
单格金额 = 网格预算 / 10    ← 一轮约 10 个买卖动作
金字塔倍数: -3% → 1.0x, -6% → 1.5x, -9% → 2.0x, -12% → 2.5x, -15%+ → 3.0x
```

### 1.6 退出规则一览

### v1.6（救火：状态机死循环 + 订单泥石流）

v1.5 的策略层改造效果如预期（硬止损次数下降、候选池更广、退出 b 增加），但 log5.txt 暴露了两条**底层一致性 bug**，必须先修才能继续做收益层优化。

#### Bug A：退出 d 减仓 50% 后 high_water 残留 → 退出 c 死循环 81 天

事故现场（华新水泥 600801，2015-09 建仓，持有至 2021 年）：

```

```

**bug 链**：

1. 退出 d 浮盈分支调 `safe_order_value(stock, -pos.value * 0.5)` 但**没清** `high_water_pnl_pct`（仍为 204.2%）和 `grid_total_buy/sell_value`
2. 网格继续按旧 `grid_target_value` 卖出，把剩余持仓全卖光
3. 持仓变成 0 后 `pos.price = 0`、`avg_cost = 0` → `pnl_pct = 0`，但 `high_water_pnl_pct` 仍 204.2% → `drop_from_peak = 204.2% - 0% = 204.2% ≥ 25%` 永远满足
4. 调 `safe_close_all` 但持仓为 0 → 聚宽返回 `下单失败，初步检查下单数量为 0`，函数返回 None → `phase` **不切换**
5. 第二天主循环又走到 GRID_RUNNING → 又触发 → 又失败 → 死循环 81 天

**修复方案（多层防御）**：

```python
# 1) 退出 d / 退出 b 减仓 50% 分支必须重置高水位 + 网格累计 + 防抖字段
if safe_order_value(stock, -pos.value * 0.5) is not None:
    st['first_buy_date'] = context.current_dt.date()    # 退出 d 才有
    st['high_water_pnl_pct'] = 0.0
    st['grid_total_buy_value'] = 0.0
    st['grid_total_sell_value'] = 0.0
    st['grid_buy_count'] = 0
    st['grid_sell_count'] = 0
    st['pyramid_levels_done'] = []
    st['exit_c_last_action_date'] = None
    st['exit_c_arm_high'] = 0.0
    st['last_grid_price'] = None  # 让 run_grid 用当前价重建基准

# 2) check_exit_c_high_water 入口加持仓 0 守卫
def check_exit_c_high_water(context, stock, st):
    pos = context.portfolio.positions.get(stock)
    if pos is None or pos.total_amount <= 0:
        return False                               # 持仓清零, 任何 high_water 残留都跳过
    ...

# 3) 主循环 BUILDING/GRID_RUNNING 阶段持仓=0 → 强制重置为 IDLE
if st['phase'] in ('BUILDING', 'GRID_RUNNING'):
    pos_chk = context.portfolio.positions.get(stock)
    if pos_chk is None or pos_chk.total_amount <= 0:
        log.info('[%s] 状态自愈: phase=%s 持仓=0 → 重置为 IDLE' % (stock, st['phase']))
        g.state[stock] = init_state()
        st = g.state[stock]
```

三层防御：(1) 减仓时正确清状态；(2) 即使漏了，exit c 也不会再触发；(3) 即使 (1)(2) 都漏，主循环也会自愈。这样任何未来的代码错误都不会再卡 80 天。

#### Bug E：高价股退出 c "切碎"——`safe_order_value` 200 股门槛漏判

**bug 链**：

1. 002311 收盘价 ~60 元 → `safe_order_value` 内 `min_value = max(2000, 60 × 200) = 12000 元`
2. 减仓 50% 时 `sell_value = pos.value * 0.5 ≈ 5715 元 < 12000 元` → `safe_order_value` 静默 return None
3. v1.7 在 `check_exit_c_high_water` 加的兜底判定是 `if sell_value < CONFIG['min_order_value_yuan'] (2000)`，**5715 > 2000 → 不走全清分支**，反而走"减仓 50%"分支调 `safe_order_value`
4. `safe_order_value` 静默 return None → `if safe_order_value(...) is not None: ...` 状态重置块跳过 → `high_water_pnl_pct` **永远不被重置**
5. 第二天 high_water 仍 ≥ `exit_c_high_water_min_profit (0.50)` → 又触发减仓 → 又静默跳过 → 死循环 30 天

```python
def calc_min_order_value(stock):
    """与 safe_order_value 内部一致的最小订单金额门槛 (v1.8 引入)."""
    cd = _get_cd(stock)
    if cd is None or cd.last_price is None or cd.last_price <= 0:
        return float('inf')   # 拿不到价格 → 不要发单
    return max(CONFIG['min_order_value_yuan'],
               cd.last_price * CONFIG['min_order_shares'])

# check_exit_c_high_water / _d_time / _b_limit_up_open 三处统一改为

    log.info('[%s] 退出 c: 高水位回撤..., 剩余仓位 %.0f 元 < 门槛 %.0f → 直接全清',
```

```python
def safe_order_value(stock, value, context=None):
    ... (原有 guard) ...
    # v1.8: 买入前预扣 available_cash, 防止同日资金竞争
    if value > 0 and context is not None:

        reserve = context.portfolio.total_value * CONFIG['min_cash_reserve_pct']
        usable = cash - reserve
        if usable < min_value:

        if value > usable:

            if scaled < min_value:
                log.info('[%s] 资金紧张: 单笔 %.0f → %.0f 仍不足门槛, 跳过'

                return None
            log.info('[%s] 资金紧张: 单笔 %.0f → 缩减到 %.0f (95%%可用)'
                     % (stock, value, scaled))
            value = scaled
    ... (原有下单 + 异常处理) ...
```

**bug 链**（v1.8 的「退出 c 强制 30 天 COOLDOWN」与 `try_exit_cooldown` 解除条件错配）：

1. **v1.8 修复 #2**：`check_exit_c_high_water` 减仓 50% 成功后，把 `phase = COOLDOWN` 且 `cooldown_until_date = today + 30 天`，目的是让 high_water 自然消退、打破 002311 类切碎循环。

#### 3.6.4 Bug I（v1.9 新发现，已修复，待 log10 验证）：`positions.get()` 引爆 WARNING 噪音

##### 现象

log9 共 **2022 条 WARNING**（v1.8 仅 71，**+27 倍**），全部一种格式：

```
WARNING - Security(code=002714.XSHE) 在 positions 中不存在,
          为了保持兼容, 我们返回空的 Position 对象,
          amount/price/avg_cost/acc_avg_cost 都是 0
```

涉及股票 Top10：002714 (516) / 600216 (466) / 603799 (366) / 002430 (192) / 300498 (151) / 600585 (138) / 600141 (62) / 603379 (34) / 688268 (30) / 002258 (29)。**全部都是经历过"全清后长期 COOLDOWN，等 score 回到 65"的票**。

##### 根因

v1.9 修 Bug H 时在 `try_exit_cooldown` 入口新加了：

```python
pos = context.portfolio.positions.get(stock)
has_position = pos is not None and pos.total_amount > 0
```

但**聚宽的 `portfolio.positions.get(stock)` 不是普通 dict**：当 `stock` 不在持仓字典中时：
- 不返回 `None`，而是返回一个空 Position 对象（amount/price/avg_cost 全 0）
- 同时打一条 WARNING 提醒"为了保持兼容"

对于全清后处于 COOLDOWN-路径 C 的票（持仓为零，等 score≥65 重置为 IDLE），cooldown_until_date 早已过期但 score 一直 < 65 → **每个交易日 try_exit_cooldown 都会调用一次 .get() → 触发一条 WARNING**。002714 全清后冻结 829 个交易日 ≈ 516 条 WARNING，与统计数据完全吻合。

##### 修复（已完成）

抽 helper 函数 `_safe_get_position(context, stock)` 用 `in` 操作符先检查存在性：

```python
def _safe_get_position(context, stock):
    """v1.9 Bug I 修复：用 in 检查避开聊宽 positions.get() 的兼容 WARNING."""
    try:
        if stock in context.portfolio.positions:
            return context.portfolio.positions[stock]
    except Exception:
        pass
    return None
```

并替换两处调用点：
- `try_exit_cooldown` 入口（v1.9 新增的判定路径，是 WARNING 主源）
- `daily_signal_and_trade` 主循环的状态自愈守卫（次要源，对 BUILDING/GRID_RUNNING 持仓 0 异常的检测路径）

##### 影响范围与等级

- **不影响策略行为或 P&L**：全部 2022 条都是日志噪音，不改变任何交易决策
- **影响日志可读性**：WARNING 区淹没了真正可疑的告警；建议 log10 把 WARNING 数检查作为常态项
- **风险等级**：低 — 这是聊宽框架自身的"兼容路径提醒"，但用 `in` 优先访问是更鲁棒的写法；其他用 `.get()` 的位置（`run_grid` / `check_exit_b/c/d/hard_stop` / `get_position_pnl_pct`）都在 GRID_RUNNING 阶段调用，stock 必然在持仓中，不需要修改

#### 优 2：动态 T0 分数阈值（解决 2015 信号荒）

现在 T0 阈值是绝对值 70，但因子在不同年份分布差异大：

```
2015 候选池: max=80.1 / median=51.7 / min=6.7   ← 仅极少数能到 70
2024 候选池: max=80+ / median=55+
```

**建议**：每天开盘后用候选池 top 10% 作动态阈值，clamp 到 [65, 75]：

```python
def compute_dynamic_t0_threshold(context):
    scores = [v for s in g.quality_universe
              for v, _ in [calc_bottom_score(context, s) or (None, None)]
              if v is not None]
    if len(scores) < 5:
        return CONFIG['bottom_score_threshold_t0']
    return float(np.clip(np.percentile(scores, 90), 65.0, 75.0))
```

实现注意：`calc_bottom_score` 内部已有 snapshot 缓存，但目前不是按日期 key 化。需要在 `before_trading_start` 清空 `g.last_score_snapshot`，让 `try_enter_t0` 复用同日缓存避免双倍 IO。

#### 优 3：退出 d "一刀切" 替代 50% 减半

**现状反思**：log7 中 600031 在 2024-04-02 第一次退出 d 减半 50% 后，剩余仓位被退出 c 一路砍 + 2025-12-03 触发"超小持仓清算"清掉。换言之 **退出 d "持仓 750 天浮盈 → 减半 50%" 这个设计其实没真正"退出"，只是把仓位切一半继续跑，最终被超小持仓清算兜底**。这跟"3 年到期评估"的设计意图相比效率很低。

**建议**：把退出 d 改为"3 年到期一刀切"：
- 浮盈/浮亏均 → **全清 + 60 日 COOLDOWN**
- 让资金完整释放，下一轮 T0 重新评估候选池

这样 600031 类长牛股在第一次 750 天就一次性退出（已赚 2.5%~200%+），剩下的 2~3 年由 T0 重评（如果分数仍 ≥70 会再次建仓）。**实际上这是 v1.8 的修复 #4 探索方向**，建议在 v1.8 修 bug 后单跑一次对照回测。

#### 优 4（log7 新增）：退出 c 切碎根因深挖 — pnl_pct 与 high_water 的"假死循环"

002311 切碎 30 次的**根因**不仅是 bug E（兑底判定漏），还有一个**模型缺陷**：

```
减仓不改 avg_cost
→ pos.value 减半，但 next bar 的 pnl_pct = (price - avg_cost) / avg_cost 不变
→ next bar high_water_pnl_pct 立即恢复到减仓前的水平
→ 任何 15% 回撤都触发减仓
→ 重复 N 天
```

理论上 v1.6 的 `arm_gap=10%` 防抖应该挡住，但实操中：
- 减仓后 high_water 设为当前 pnl_pct（设 33%）
- 第二天 price 反弹 → pnl_pct 恢复到 35% → high_water 跳到 35% > arm_high (33%) + gap (10%) = 43%? 不！这个公式是 `high_water < arm_high + gap` 才静默，**high_water = 35% < 43% → 应该静默**
- **真相**：这个公式只在某些边界 case 失效。002311 的实际触发更可能是 bug E 的"减仓未成交 → high_water 永不重置"

**根本修复**（v1.8 优 0-3 已涉及）：减仓后强制 COOLDOWN 30 天，**绕过 high_water/pnl 的所有计算**。这是最简单可靠的设计。

**进阶建议**：考虑把 high_water 改为基于"绝对价格"而非 pnl_pct：

```python
st['high_price_after_buy'] = max(st.get('high_price_after_buy', 0), last_price)
drop_from_peak_price = 1 - last_price / st['high_price_after_buy']
if drop_from_peak_price >= 0.15:
    ...
```

这样减仓不影响 high_price，状态更直观，且不会因为 avg_cost 不变导致"假反弹"。

### 4.2 中优先级

#### 优 5：网格预算随价格扩张（缓解 302 次「网格已达上限」）

**现状**：`grid_target_value = base * 0.5` 在 T2 完成时固定。当股价从 8.6 涨到 19.4（如 600801），底仓市值翻倍，但网格预算还是当年的 37510 元。log7 仍有 302 次"网格已达上限"，主要集中在大牛股的中后期。

**方案 A（保守，推荐）**：完全不扩。退出 b 周期顶全清（v1.7 仍是 4 次）+ v1.7 超小持仓清算共同把大牛股清掉，资金腾出来给新底部信号。

**方案 B（激进）**：每天 `grid_target_value = max(初始预算, base_target_value × 当前价/T0价 × 0.3)`，给上限避免无限膨胀。

**方案 C（折中）**：在涨幅达 +50%/+100%/+150% 时各自扩 +25% 网格预算（共 +75%）。

> 优 3（退出 d 一刀切）+ 优 6（行业大类互斥）+ 方案 A 三者结合，可能完全消除"网格已达上限"问题，且不需要复杂的预算扩张逻辑。

#### 优 7：网格步长 ATR 估值修复

观察到 600031 的网格步长出现 `步长=3.54%/3.58%/3.61%/3.63%` 持续上升，说明 ATR 跟着价格波动放大。但 base_grid_chunk 是 T2 时定的固定金额，导致"步长越大、金额越固定"的失衡 — 高波动时反而 sell 成交体量过小。

**建议**：base_grid_chunk 也应跟随当前价漂移（按比例）：

```python
chunk_value = base_grid_chunk * (last_price / first_buy_price)   # 价格涨1倍, 单格金额涨1倍
```

效果：高位震荡时网格成交体量自然扩大，更容易突破 min_order_value 门槛。

#### 优 8：科创板可平仓数量精算

```python
if value < 0:
```

#### 优 9：T0 信号收敛诊断（log7 转化率回升但绝对数仍偏低）

v1.7 的 T0 转化率 8.7%（v1.6 是 4.27%）虽显著回升，但 586 个累计天数中只有 51 个真正落地建仓。**剩下 535 个"达标 5 天但没建"的天数都被哪些原因拦下来？**

#### 优 16：考虑放弃 50% 减半作为通用退出动作（v1.9 后再评估）

**回顾连续 5 轮和"减半"相关的 bug**：

| 救火版本 | bug 现场 | 根因 |
|---|---|---|
| v1.6 | 600801 退出 c 全清 81 天 | 减半后 high_water 残留 + 持仓 0 触发死循环 |
| v1.7 | 600031 退出 d 反复触发 3 天 | 减半金额过小 → safe_order_value 静默跳过 + first_buy_date 未重置 |
| v1.7 残留 | 47 条 "开仓数量小于 200" ERROR | min_order_shares 不够覆盖科创板 |
| v1.8 已修 | 002311 退出 c 切碎 30 次 | 减半金额 < `last × 200` → safe_order_value 静默跳过 + high_water 永不重置 |
| v1.9 已修 | **002714 减半后 cooldown 困死 916 天** | **减半 cooldown 与退出 a cooldown 共用解除条件 → 牛市 score 永不到 65** |

**5 轮 bug 都和"减半 50%"路径耦合**。这强烈提示这个设计本身有结构性问题：

1. 减仓不改 avg_cost → next bar pnl_pct 立即恢复 → high_water 立即恢复 → 退出 c 反复触发
2. 减仓后剩余仓位金额可能跌到任何 guard 门槛之下 → safe_order_value 静默 → 状态不重置
3. 减半后 phase 仍 GRID_RUNNING → 继续走全部 exit 流程 → 累积新 bug 路径

**思考方向**：把退出 b/c/d 的减半都改成"**全清 + 60 日 COOLDOWN + 自动重新参与下一轮 T0**"：

```python
# v1.8+ 设计: 所有 exit 都全清, 不存在"减半续持"
def check_exit_b_limit_up_open(...):
    if ...:
        safe_close_all(stock)
        st['phase'] = 'COOLDOWN'
        st['cooldown_until_date'] = today + timedelta(days=60)
        return True

def check_exit_c_high_water(...):
    if drop_from_peak >= 0.15:           # 降低阈值, 用 30 日 COOLDOWN 兜底
        safe_close_all(stock)
        st['phase'] = 'COOLDOWN'
        ...

def check_exit_d_time(...):
    if holding_days >= 750:              # 浮盈/浮亏均全清
        safe_close_all(stock)
        ...
```

**优点**：
- 模型大幅简化（少了 BUILDING/GRID_RUNNING/减半后混合状态的所有 case）
- 4 轮 bug 的根源彻底消失（没有"减半后剩余金额过小"路径）
- 资金完整释放，下一次 T0 信号可以重新获得完整建仓机会

**缺点**：
- 错过"减半续持 + 后续涨"的潜在收益
- 但 T0/T1/T2 重新建仓机制完全可以替代（log7 就有 51 次新 T0，转化率 8.7%）

**建议（v1.9 后修订）**：等 log9 验证 Bug H 修复效果后再决定。

**v1.9 期望路径**：先看 log9 是否能在保留减半设计的前提下追平 / 超过 log7 的退出 b 表现：
- 若 log9 退出 b ≥ 13 次 + 002714 重新拿到 +180% 量级 → **保留减半设计**（v1.9 已用最小改动解决了 Bug H）

#### 优 17（log8 新增）：建仓死区超时回退（解决 000932 类长尾死仓）

**事故现场（华菱钢铁 000932，log8）**：

```
2019-11-15 T0 建仓 4.20 元 / 目标=28267 / 本次=4240
            ← 此后股价在 4 元附近窄幅波动 5 年
2024-09-03 T1 加仓 (左侧, 跌幅 9.52%)   ← 5 年才触发 T1!
2024-09-04 T1 加仓 ...                    ← 又 T1?(说明状态仍 BUILDING tier_filled=1)
2024-09-10 T2 加仓 (左侧)
2024-09-11 T2 加仓 ...                    ← 终于 GRID_RUNNING
2024-09-12 退出 d: 持仓 1763 天浮亏 -0.0% → 全部清仓   ← 第二天就被 750 天评估清掉
```

5 年 BUILDING 状态期间：(1) `single_stock` 配额一直被占用 4240 元；(2) 资金锁死无法用于其他票；(3) 最后浮亏 0% 退出 d 全清，等于 5 年时间机会成本完全损失。

**建议**：在 `try_build_t1_t2` 加超时回退：

```python
# CONFIG 新增
'tier_t1_timeout_days': 365,    # T0 后 365 天仍未达 T1 → 自动 IDLE 释放配额
'tier_t2_timeout_days': 730,    # T0+T1 后 730 天仍未达 T2 → 自动 IDLE

# try_build_t1_t2 入口
days_since_t0 = (context.current_dt.date() - st['first_buy_date']).days
if st['tier_filled'] == 1 and days_since_t0 > CONFIG['tier_t1_timeout_days']:
    log.info('[%s] T1 超时回退: T0 后 %d 天未达 T1 触发条件 → 全清 + IDLE',
             stock, days_since_t0)
    safe_close_all(stock)
    g.state[stock] = init_state()  # 重置整个状态机
    return
```

**预期收益**：000932 类窄幅震荡票在 365 天后自动释放配额，让真正能跑起来的票占用资金。注意：**不要套用 60 天 COOLDOWN**，这种票"不动"不是因为高位回撤，而是单纯没机会，IDLE 后任何时候分数 ≥ 70 都能立即重新 T0。

**风险**：参数调过激进（如 180 天）可能在熊市底部 V 反前夕把人赶出，建议从 365 天起试。

## 7. 设计哲学回顾

回到你最初的 4 句话：

> 1. 自动选优质周期股
> 2. 周期底部买 1/2 底仓
> 3. 3% 网格
> 4. a) 网格卖完退出 b) 涨停打开退出

每条都被放大成一个工程问题：

| 原意 | 工程化拆解 |
|---|---|
| "优质周期股" | 4 维度财务过滤 + 13 行业 36 标的硬编码（v1.5 扩，避免动态选股带来的回测样本污染）+ 硬止损惯犯黑名单（v1.5）|
| "周期底部" | 4 因子（PB/回撤/OCF/RSI）打分 + 5 日持续达标 + 三档分批建仓 + T0 防幽灵机制（v1.2）|
| "3% 网格" | 动态步长（避开过密/过疏）+ 不对称（跌买更密）+ 金字塔（越跌越多）+ 最小订单 guard（v1.6/1.7）|
| "网格卖完退出" | 退出 a：保留底仓 + 60 日冷却 |
| "涨停打开卖出" | 退出 b：盈利门槛 + PB 分位区分常规 / 周期顶（**log7**：13 次、均 60.7%；**log8**：11 次、均 49.9%；**log9**：11 次、均 **59.9%**、最大 +184.6% — Bug H 修复后 002714 大牛全部恢复，详 §3.6.2/§3.6.3）|

你给的是策略灵魂，工程化加了"如何防自己手抖、防数据噪声、防熊市底部 V 反、防状态机死循环"等 robust 层。**多版本连续验证退出 b 核心规则有效**；**v1.8 (log8)** 在 **ERROR 归零、无 002311 切碎** 的前提下暴露 **Bug H**（休眠与解除条件错配，牛市 002714 冻结 2.5 年）→ **v1.9 (log9)** 修复 Bug H，**002714 +184.6% 完整恢复，路径 B 触发 31 次，末日 GRID_RUNNING 1→6**；同时暴露 **Bug I**（聊宽 `positions.get()` 对全清股票的兼容 WARNING，2022 条噪音），同代补丁加 `_safe_get_position` helper → **v1.9 (log10) 封板基线**：**0 ERROR / 0 WARNING / 0 RuntimeWarning**，与 log9 行数差恰好等于 2022 条 WARNING，30+ 项交易指标一字不差，零副作用。

**核心教训（v1.9 升级版）**：

1. **下单接口的"成功"和"静默跳过"必须有不同的语义**：策略不能把 `None` 当成"等下次再试"，要么明确传播错误、要么明确兑底处理 — v1.8 把 `safe_order_value` 内部的 min_value 算法暴露成独立函数，让所有调用者用同一套门槛判定
2. **状态字段的重置应该绑定到状态转换边界**：`first_buy_date / high_water / grid_total_*` 等应该在 phase 切换时统一处理，而不是分散在多个 exit 函数里。v1.8 强制 `phase = COOLDOWN` 后，状态自然由 `try_exit_cooldown` 兜底重置
3. **"减半续持"是 bug 温床**：连续 4 轮 bug 都和"减半 50% 路径"耦合，v1.8 用强制 COOLDOWN 大大减弱了"减半后还在 GRID_RUNNING"的 bug 路径；但 v1.8 又引入了 Bug H — **设计教训：不同来源的 cooldown 不能共用解除条件**。退出 a 的 cooldown 是"高位卖完了等回落再入"，退出 c 减半的 cooldown 是"重置 high_water 防切碎"，两者的语义完全不同。**v1.9 用「`last_high_after_grid` 是否设置」隐式区分**，比新增 `cooldown_reason` 字段更小入侵
4. **同日资金竞争是聚宽撮合层的隐藏行为**：v1.7 的 `min_order_value/shares` 都是单笔判定，没考虑全组合层面的现金竞争。v1.8 在 `safe_order_value` 入口预扣 `available_cash` 才能根治
5. **每个版本都会暴露上一版本的"隐性副作用"**：v1.7 修了 v1.6 的金额 guard 副作用 → v1.8 修了 v1.7 的高价股门槛漏判 → v1.9 修了 v1.8 的 cooldown 解除条件错配 → v1.9 修 Bug H 时调用 `positions.get()` 又触发了聊宽兼容路径的 WARNING（Bug I）。**回测复盘必须看"被路径剥夺的潜在收益"** + **WARNING 异常激增也要追到根因**，不能只盯着 ERROR
6. **聊宽 `portfolio.positions.get(stock)` 不是普通 dict 行为**：对不存在的 stock 返回空 Position + 打 WARNING（"为了保持兼容"）；这种"看似无害"的 API 设计在循环调用时会爆炸式产生噪音。**鲁棒模式**：用 `stock in context.portfolio.positions` 先检查；或封装 `_safe_get_position` helper 集中处理
7. **指标对比"看起来没改善"≠ 没价值**：log9 退出 b 总数仍是 11（vs log8 的 11），但**质量大变**（最大浮盈 85.7% → 184.6%、平均 49.9% → 59.9%）。**总数指标对极端值不敏感**，要用平均/分位数/最大值多维度评估

> **v1.9 当前状态：✅ 已封板**。Bug H + Bug I 两个 bug 均已修复且经 log9/log10 双重回测验证。log10（封板基线）= 20255 行 / 0 ERROR / 0 WARNING / 0 RuntimeWarning，全部 30+ 项交易行为指标稳定，002714 +184.6% 退出 b 完整复现。**下一步关键决策**：v2.0 第一阶段重点投入 **优 1 (trade_log)** 让后续实验从"人眼 grep 20000 行"升级到"DataFrame 归因"；中期决策点是 **优 16（全清替代减半）vs v1.9 现状** 的对比 A/B 回测，需要 trade_log 数据做归因评估。
